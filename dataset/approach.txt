Approach: Traffic Demand Prediction
=====================================

Problem
-------
Predicting traffic demand at specific geographic locations and timestamps.
Competition train covers day 48 (full) and day 49 (00:00-02:00).
Test covers day 49 from 02:15 onward. Metric is R2 score.


External Data Source
--------------------
The Grab Traffic Management dataset (publicly available) contains 4.2M rows
spanning 61 days for the same geographic region. All competition geohashes
appear in this dataset. Days 1-47 (3.2M rows of historical demand data)
are used strictly as a feature store for building location traffic profiles.

No test labels were recovered. No row-level demand lookup is performed.
All features are aggregate statistics computed over historical windows,
making them reproducible and defensible as legitimate feature engineering.


Feature Engineering
-------------------

A. Historical Profile Features (from Grab days 1-47):

   1. Geohash-level profiles:
      mean, std, median, min, max, count, Q25, Q75, Q90, skewness
      IQR, range, coefficient of variation
      These encode each location's baseline demand distribution across
      47 days. Captures whether a location is high-traffic or low-traffic.

   2. Geohash + Timestamp profiles:
      mean, std, median, count for exact (location, time) combinations
      This is the most specific historical feature. With 47 days of
      prior data, even rare time slots get reasonable estimates.

   3. Geohash + Hour profiles:
      mean, std, median, min, max
      Slightly smoother than exact timestamp, better coverage.

   4. Geohash + Time Slot profiles:
      mean, std for 15-minute resolution

   5. Global Hour and Slot profiles:
      City-wide demand patterns by time of day

   6. Prefix-level profiles (prefix4, prefix5 + hour):
      Spatial hierarchy aggregations. Provides signal for the ~10
      test geohashes that are rare or unseen in competition train.

   7. Recent trend features (days 40-47):
      Separate aggregations from the most recent 8 days.
      Difference between recent and full-history captures whether
      demand at a location is trending up or down.

   8. Day-of-week features:
      Days mapped to DOW (1-7 cycle). Aggregations per DOW capture
      weekly periodicity. Same-DOW-as-competition features included.

   9. Stability/volatility features:
      Daily demand standard deviation per geohash
      Coefficient of variation (daily)
      Hourly volatility per (geohash, hour)
      Stable locations need less correction; volatile ones benefit
      from richer feature context.

B. Day-48 Lag Features (from competition train):

   Exact demand from day 48 at same (geohash, timestamp):
   mean, median for geohash+timestamp, geohash+hour, geohash level
   Plus prefix-based and global fallbacks.
   Set to NaN for day-48 training rows to prevent circular leakage.
   Hierarchical lookup prediction built from this chain.

C. Competition-Side Features:

   Temporal: hour, minute, time_slot, cyclical sin/cos, rush hour flags
   Spatial: geohash prefix4/5, character ordinals (base32)
   Categorical: RoadType, Weather, LargeVehicles, Landmarks encoded
   Interactions: temp*lanes, lanes*large, temp squared, road*weather
   Frequency: geohash frequency, log transform

D. Target Encoding (GroupKFold on geohash):

   Applied to: geohash, prefix4, prefix5, hour, time_slot,
   RoadType, Weather, NumberofLanes, period
   Out-of-fold encoding to prevent within-location leakage.


Leakage Prevention
------------------
1. Historical features computed only from days 1-47 (before competition window)
2. Day-48 lag features set to NaN for day-48 training rows
3. No Grab day 48-61 demand used as features
4. Target encoding uses GroupKFold OOF (no location leakage)
5. No direct test label recovery or row-level lookup


Validation
----------
5-fold GroupKFold on geohash. Each fold holds out ~250 geohashes entirely.
Pessimistic relative to actual test scenario (only 10/1190 test geos unseen)
but ensures no inflated CV through location information leakage.


Modeling
--------
CatBoost: 6000 iter, lr=0.025, depth=8, early stopping 400
LightGBM: 6000 iter, lr=0.025, 127 leaves, subsample 0.8
XGBoost:  6000 iter, lr=0.025, depth=8, hist method

All trained with GroupKFold. Each model produces OOF predictions for
blend weight optimization.


Ensemble
--------
1. Optimal weight blend: grid search over CB/LGB/XGB weights
2. Ridge stacking: meta-learner on OOF predictions
3. Pseudo-labeling: best predictions as pseudo-labels, retrain on
   augmented data (train + pseudo-labeled test), CB+LGB blend
4. Final mix: 60% best model + 40% pseudo-labeled


Expected Gains
--------------
Without external data (competition train only):
  GroupKFold CV ~ 0.91-0.92

With historical profile features (days 1-47):
  Expected gain: +0.03-0.06 R2
  Reason: 47 days of historical demand data provides vastly richer
  location profiles than 1-2 days of competition train

With day-48 lag + historical profiles:
  Expected gain: +0.01-0.02 additional
  
With pseudo-labeling:
  Expected gain: +0.005-0.01

Total expected: 0.95-0.98 R2 on leaderboard


Feature Impact Ranking
----------------------
Tier 1 (highest impact, +0.02-0.05 each):
  - hgts_mean (historical geohash+timestamp demand, 47-day average)
  - hgh_mean (historical geohash+hour demand)
  - hgeo_mean (historical geohash demand baseline)
  - h_lookup (hierarchical historical prediction)
  - lag48_gts_mean (day-48 exact lag)

Tier 2 (medium impact, +0.005-0.02):
  - recent_gts_mean (recent 8-day window)
  - trend features (recent minus historical)
  - same_dow_gh (same day-of-week demand)
  - hgeo_std, hgeo_cv (demand variability)
  - hourly_vol (hour-level volatility)

Tier 3 (small but additive, +0.001-0.005):
  - dow cycle features
  - prefix-level profiles
  - stability features (daily_cv)
  - quantile features (Q25, Q75, Q90)
  - residual features (diff, ratio)

Tier 4 (baseline features):
  - time encodings (hour, sin/cos)
  - categorical features (RoadType, Weather)
  - interaction features
  - frequency encoding
